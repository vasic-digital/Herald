QA-ANCHOR: HRD-133-TGRAM-TOKEN-REDACTION-20260528

HRD-133 — Token-redaction symmetry (tgram adapter)
====================================================

Mandate
-------
Bring the tgram adapter up to the slack adapter's §107 redaction bar.
The Telegram Bot API token (format "<bot_id>:<auth_secret>", e.g.
"123456:supersecret") MUST NEVER appear in any error message that
Send / SendReply returns to the caller.

Implementation
--------------
- Added commons_messaging/channels/tgram/errors.go:
    func sanitizeTgramError(msg, token string) string
  Mirrors qaherald/internal/messenger/slack.go:327 (the slack adapter's
  sanitizeError) — same shape, same semantics: empty token = passthrough;
  otherwise strings.ReplaceAll(msg, token, "[REDACTED]").

- Wrapped every error-returning path in Send + SendReply
  (commons_messaging/channels/tgram/send.go) with sanitizeTgramError:
    Send:
      - ensureBot() failure
      - strconv.ParseInt(chatID) failure
      - bot.Send failure (via withRetry429)
    SendReply:
      - ensureBot() failure
      - bot.Send (text) failure (via withRetry429)
      - bot.Send (attachment fan-out) failure per attachment (via withRetry429)
  These are the SAME sites that previously used fmt.Errorf with %w —
  they now use %s + sanitizeTgramError(err.Error(), a.botToken).
  Empty error chain is the cost; the security guarantee is the gain.

- Defense-in-depth note: telebot itself already redacts /bot<token>
  path occurrences in lower-level wrapError (submodules/telebot/
  errors.go:282 — redactToken regex). Herald's layer is the BARE-TOKEN
  scrub for any code path that interpolates the raw token without the
  /bot prefix.
