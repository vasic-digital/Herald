QA-ANCHOR: HRD-134-TGRAM-RATE-LIMIT-429-20260528

HRD-134 — Bot API rate-limit (429 + retry_after) honored
=========================================================

Mandate
-------
When the Telegram Bot API returns 429 Too Many Requests with
parameters.retry_after = N, tgram.Send MUST sleep N seconds and retry
(bounded) instead of returning the error immediately. Spec V3 §17
"rate-limit aware" + tgram-completeness mandate (2026-05-28 live-gap
analysis, Section 2 task #1).

Implementation
--------------
- Added const maxRetry429Attempts = 3 and maxRetryAfterSeconds = 60
  in commons_messaging/channels/tgram/send.go.

- Added withRetry429(ctx, token, sleeper, fn) helper:
  Calls fn(); on FloodError + retry_after extractable → sleep clamped
  retry_after honoring ctx.Done() → retry. Caps at maxRetry429Attempts
  total invocations. Caps per-attempt sleep at maxRetryAfterSeconds.
  Logs each retry with the retry_after value, the clamped value, and
  the attempt counter (and sanitizes the token from the log line).

- Added realSleep(ctx, d) — production sleeper that blocks with a
  ctx-aware timer.

- Added Adapter.sleeper field (commons_messaging/channels/tgram/
  tgram.go) — injection seam for test fakes; nil = realSleep default.

- Added extractRetryAfter(err) helper (errors.go) — single point
  where telebot.FloodError → seconds mapping is asserted.
  Uses errors.As against telebot.FloodError value (NOT pointer; per
  submodules/telebot/errors.go:17 FloodError is a struct value type
  with non-pointer Error() method).

- Wrapped all three telebot bot.Send call sites in withRetry429:
    Send: text body
    SendReply: text body
    SendReply: attachment fan-out (per attachment — each independently
    rate-limit-aware)

retry_after extraction method
-----------------------------
Used telebot's typed FloodError (NOT direct body parse).
  submodules/telebot/errors.go:17 — type FloodError struct {
      err        *Error
      RetryAfter int
  }
  submodules/telebot/bot_raw.go:329 — FloodError is constructed when
  extractOk sees error_code=429 + parameters.retry_after in the JSON
  envelope.
extractRetryAfter uses errors.As(err, &fe) with fe a FloodError value
to recover the RetryAfter int field.

§107 anti-bluff anchor
----------------------
The sleeper-injection seam (Adapter.SetSleeperForTest) is the test-only
hook that records the requested durations. The four tests assert
DURATION VALUES directly — a regression that:
  - drops the withRetry429 wrap → HonorsRetryAfter fails (gets the 429 immediately)
  - drops the 3-attempt cap → GivesUpAfter3Attempts loops forever or fails count
  - drops ctx.Done() handling → RespectsCtxCancel times out at 30s
  - drops the clamp → ClampsRetryAfter_60s sees a 300s request

All four failures are deterministic and fast (each test runs in <50ms
with the fake sleeper).

Status: HRD-134 IS FIXED — tgram.Send + tgram.SendReply now honor Bot
API 429 retry_after, bounded at 3 attempts × 60s clamp = 180s max
wall-clock for the worst-case stuck-in-rate-limit case. End users no
longer see Send fail on the first 429 — the adapter waits and retries
transparently.
