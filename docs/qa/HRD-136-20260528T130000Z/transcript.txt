CONTEXT
=======
Spec V3 §11.1 says Telegram inbound CAN use webhooks OR getUpdates long-poll.
Wave 6 shipped getUpdates (pherald listen, subscribe.go). HRD-136 lands the
cryptographic-validation layer for the future webhook ingress path — so when
a V1.x operator switches to webhooks, the secret_token impersonation defense
is already proven by hermetic tests.

Bot API §6 (https://core.telegram.org/bots/api#setwebhook) stamps every webhook
POST with header `X-Telegram-Bot-Api-Secret-Token: <secret>` (set via
setWebhook(secret_token=…)). Validating that header constant-time on every
request is mandatory — without it an attacker who guesses the webhook URL
can impersonate Telegram. subtle.ConstantTimeCompare denies the timing-
oracle path that would let an attacker recover the secret one byte at a time
via naive `==` short-circuit.

DELIVERABLE
===========
NEW FILE: commons_messaging/channels/tgram/webhook.go
  - WebhookHandler struct + NewWebhookHandler(secret, handler, logger) constructor
  - http.Handler.ServeHTTP that:
      1. enforces POST (405 on mismatch)
      2. constant-time-compares X-Telegram-Bot-Api-Secret-Token header
         against the configured secret using subtle.ConstantTimeCompare
         (401 on mismatch — uniform empty-body, no token leak)
      3. decodes the body into telebot.Update (400 on malformed)
      4. converts Update → commons.InboundEvent via updateToInboundEvent
      5. dispatches to the caller-supplied commons.InboundHandler
         (500 on handler error — error message logged, NEVER echoed)
      6. returns 200 with body `{"ok":true}` per Bot API convention
  - Refuses empty secret at construction time with a loud error.
  - Refuses nil handler at construction time with a loud error.
  - subtle.ConstantTimeCompare is invoked at webhook.go:222 + :229.
  - HeaderName() exposes the canonical header string for tests + future
    `pherald listen --webhook` wiring.

ANTI-BLUFF CHECKS
=================
- subtle.ConstantTimeCompare is on the hot path (webhook.go line 222 + 229).
  TestWebhook_ConstantTimeCompare directly invokes secretMatches() against
  equal-length-but-different-content secrets (`AAAA...A` vs `AAAA...B`)
  to prove the comparator does not regress to a `==` short-circuit
  (which would compare byte-by-byte in non-constant time and leak the
  secret via timing).
- 401 path returns EMPTY body — TestWebhook_InvalidSecret_Returns401_NoTokenLeak
  asserts response body contains NEITHER the configured secret NOR the
  presented (probe) token; TestWebhook_ConstantTimeCompare asserts
  len(body) == 0 on every 401, denying the body-grep oracle.
- Constructor refuses empty secret AND nil handler — TestWebhook_EmptySecretAtConstruction_ErrorsLoud
  asserts both rejection paths return non-nil error + nil handler.
- Decode-error trace lands in logger only, never response body —
  TestWebhook_MalformedJSON_Returns400 asserts response body does NOT
  contain "decode" AND logger buffer DOES contain "decode failed".
